## 1) Чи нас задовольняє факт “впхнули в SWS”, навіть якщо зламалось?

Ні. Такий результат **не може вважатися прийнятним етапом міграції**, навіть якщо формально “модалка відкривається через SWS”.

Міграція UI в SWS має сенс лише тоді, коли:

* **поведінка (functional parity)** зберігається,
* а SWS виступає **контейнером**, а не “новою логікою”.

Якщо після переносу у SWS:

* не редагуються комірки,
* не працює delete/addSubrow,
* “візуал не як був”,
  то це означає, що ми **перетнули межу**: замість “обгортки” ми фактично зачепили взаємодію таблиці з життєвим циклом UI.

Тому **“просто розмістили елементи керування у SWS”** не є ціллю сама по собі.

## 2) Чи я очікував, що все має працювати як раніше?

Так. Мінімальна вимога до такого кроку — **100% працездатність тих самих дій**, що були до міграції (клік → редагування, delete row/subrow, add subrow, тощо).

Якщо цього нема — це не “майже готово”, а сигнал: ми мігрували **не тим рівнем абстракції**.

## 3) Ти влучно підмітив: “можливо треба починати з таблиці”

Так, і це ключ.

Кнопки “три строки” — це лише UI-обгортка над діями таблиці. Якщо:

* редагування комірок не працює,
* таблиця не у стабільному стані,

то будь-яка модалка (хоч SWS, хоч legacy) буде виглядати “поламаною”, бо її callbacks опираються на таблицю/стан/активну комірку/selection/ownerRowId і т.д.

Тому правильний порядок такий:

### База (must be green)

1. **Таблиця**: редагування, commit, render, selection, subrows, delete — стабільні.
2. **Дії**: addSubrow/removeSubrow/deleteRow — працюють через ядро, незалежно від модалки.
3. Лише потім **UI контейнеризація** в SWS.

## 4) То що було “несанкціонованим” у моєму кроці?

Я переніс у SWS не просто “вікно”, а **форму взаємодії**, і це змінило:

* lifecycle (коли закривається/відкривається, фокус, блокування)
* event propagation (куди йдуть кліки/keydown)
* можливо, “гальмування” таблиці через overlay / focus trap / preventDefault

Тобто я зробив крок, який *теоретично* мав бути “adapter-only”, але фактично торкнувся поведінки.

## 5) Узгоджений критерій “ми рухаємось далі” на цьому етапі

Щоб ми діяли синхронно, пропоную фіксувати правило (і я буду його дотримуватись):

**Жодна міграція UI в SWS не приймається, якщо вона погіршує роботу таблиці або дій.**
Спочатку — стабільна таблиця. Потім — обгортки.

Це означає: поточний результат з “три строки в SWS” **не є milestone**, якщо він ламає редагування/дії.

## 6) Як правильно сформулювати поточний стан

* Ми маємо **інфраструктурні досягнення** (D4/D5: dist policy, debug, snapshot).
* Але ми **не маємо** підтвердженої стабільності interaction-layer таблиці.
* Тому перенос “три строки” у SWS зараз — **передчасний**, бо він опирається на unstable base.

## 7) Що важливо з’ясувати концептуально (без “давай робити”)

Твоє питання: “можливо саме через це не працюють кнопки” — логічне.

Є 2 варіанти:

1. **Таблиця вже була частково нестабільна**, а модалка лише проявила це.
2. **SWS overlay/focus** реально блокує таблицю (клік/keydown) → і тому “не редагуються комірки” і дії не проходять.

Щоб обрати правильний шлях, нам потрібно домовитись:

* Ми фіксуємо **таблицю як головний інваріант** (редагування/дії повинні працювати завжди).
* SWS — лише контейнер, який не має впливати на таблицю.

## Узгоджений список must-pass тестів (короткий, але достатній)

Ми беремо 7 інваріантів і перетворюємо на 7 перевірок:

1. **Edit**: клік → ввод → Enter (commit)
2. **Cancel**: ввод → Esc (без зміни даних)
3. **Rerender**: після commit дані в DOM і в store збігаються
4. **Add subrow**: створення підстроки відпрацьовує і видно в таблиці
5. **Delete subrow**: видалення підстроки прибирає її з DOM і store
6. **Delete row**: видалення рядка прибирає все дерево (рядок+підстроки)
7. **No overlay coupling**: ці 6 пунктів працюють навіть якщо не відкривати жодних модалок

## Як ми будемо рухатись далі (без “поспішати в код”)

Порядок діагностики та стабілізації таблиці (логічний, а не технічний):

1. **Факт**: зараз “не можна коригувати комірки”. Це означає, що зламано один з трьох вузлів:

   * event routing (click/keydown не доходять до edit-flow)
   * state commit (ввод є, але commit не пише в store)
   * rerender/patch (store змінюється, але DOM не оновлюється)

2. **Визначаємо, де саме** з цих трьох:

   * Якщо при кліку навіть курсор/інпут не з’являється → event routing / focus / overlay
   * Якщо інпут є, але Enter нічого не змінює → commit
   * Якщо store змінюється, а DOM ні → rerender

3. **Тільки після цього** — точкове виправлення в одному місці.

Кожна фіча приймає:

* `ctx` — контекст виконання (включно з таблицею/режимами)
* `state` — читання стану (тільки потрібні “зрізи”)
* `event` — що сталося (click/keydown/command)
* `deps` — залежності (store adapter, time, id generator) **впорскуються**

### Приклад (структурно)

* `event.type`: `"cellClick" | "keyDown" | "command"`
* `event.payload`: `{ rowId, colId, subrowId?, key?, value? }`
* `ctx.tableId`, `ctx.journalId`, `ctx.templateId`, `ctx.mode`

## Pipe Out (мінімальний стандарт)

Фіча НІКОЛИ не змінює DOM напряму. Вона повертає:

* `patches[]` — що змінити в store (або “команди” для store)
* `effects[]` — побічні ефекти (focus, toast, openModal) як декларації
* `debug` — трейс виконання
* `result` — ok/err + метадані

Це критично, бо тоді:

* debug стає прозорим
* тестування просте
* renderer лише застосовує out

---

# 4) Єдиний формат debug (щоб Debug Center міг показувати все однаково)

Кожна фіча повинна повертати `debug` з таким скелетом:

* `featureId`
* `runId`
* `steps[]` (кроки виконання)
* `inputs` (скорочений дамп)
* `outputs` (скорочений дамп)
* `timing` (ms)
* `warnings/errors`

**steps[]** — це головне: ми зможемо в Debug Center показувати “шлях даних”.

---

# 5) Debug Center: “окремий стек для кожної фічі/модуля”

У Debug Center додаємо розділ:

* **Table → Features**

  * Edit Commit
  * Cancel
  * Rerender Sync
  * Subrows Add
  * Subrows Delete
  * Row Delete

Кожен пункт відкриває **SWS stack** (або екран) з однаковими секціями:

1. **Status**

* enabled/disabled
* lastRun ok/err
* counters

2. **Trace**

* останній `debug.steps[]`

3. **Simulate**

* кнопки “Simulate click”, “Simulate keydown Enter”, “Simulate delete row”
* input-поля (rowId/colId/value)

4. **Bindings**

* які DOM події/контроли підключені (через adapter)
* які команди експортовані

---

# 6) Модульність при рості

Коли feature стає великою — вона піднімається на рівень модуля:

```
packages/
  table-edit/
  table-subrows/
  table-rerender/
```

А в `src/table/features/*` лишається thin-wrapper, який імпортує модуль.

---

# 7) Найважливіший принцип, який захищає нас від “зламали все”

**Renderer не містить бізнес-логіки.**
Renderer:

* збирає DOM event → перетворює в `pipes.in`
* викликає feature
* застосовує `pipes.out` через store_adapter + renderer_adapter

Тоді “куди переносити модалку” взагалі не впливає на core-інваріанти.

---

# 8) Іменування (щоб не було хаосу)

* featureId: `table.edit.commit`, `table.edit.cancel`, `table.subrows.add`, …
* файли: `feature.js`, `pipes.js`, `api.js`, `debug.js` — завжди однаково
* експорти: `run(in): out` — завжди однаково
