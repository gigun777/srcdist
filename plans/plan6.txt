# 🧠 2. На чому ми зупинились

На момент step46:

## 2.1 Стан системи

✔ Backup v2 стабільний
✔ Import pipeline повністю контрольований
✔ Legacy v1 відключений
✔ Build працює (merge mode)
✔ Runtime dist assets завантажуються коректно

❌ UI нестабільний
❌ Конфлікти модалок
❌ Неконсистентний дизайн
❌ Esc працює непередбачувано
❌ Частина екранів живе на UI.modal
❌ Частина — на SWS
❌ Частина — на injected DOM

---

## 2.2 Головна проблема

Не окремі баги.

Проблема системна:

> В проекті немає одного центру керування модалками.

Маємо:

* SettingsWindow (SWS)
* UI.modal
* transfer_modals injected overlay
* різні keydown listeners
* різні стекові механіки

Це призвело до:

* дублювання
* подвійних викликів
* неконтрольованих overlay
* перехоплення подій
* втрати фокусу
* розриву логіки таблиці

---

# 📊 3. Висновки після аналізу

## 3.1 Головний архітектурний висновок

Неможливо далі розвивати систему, поки:

* не буде централізованого SWS ядра
* не буде одного контролера стеку
* не буде єдиної keyboard policy
* не буде стандартизованого ScreenSpec

---

## 3.2 Що стало очевидно

1. SWS — найкраща база для єдиного modal engine.
2. transfer_modals injection — тупиковий шлях.
3. UI.modal треба поступово вивести.
4. Кожен екран має бути описаний як декларативний spec.
5. Таблиця має мати єдиний rerender engine.

---

# 🏗 4. Стратегія, яку ми затвердили

## Глобальна ціль

> Побудувати одне ядро SWS, яке керує всіма модалками, стеком, кнопками, клавіатурою, станом.

# 🧩 6. Ключові принципи, які ми зафіксували

1. Один екран = один ScreenSpec
2. Жоден екран не створює DOM самовільно
3. Нема інжекцій без контролера
4. Нема глобальних listeners поза SWS controller
5. Нема дублювання UI.modal + SWS
6. Таблиця не залежить від модалок напряму

---

# ⚠ 7. Що ми вирішили НЕ робити

❌ Не латати transfer_modals
❌ Не додавати нові overlay
❌ Не намагатися “швидко виправити” всі баги
❌ Не чистити dist повністю до завершення UI міграції

Найбезпечніша точка входу:

**SWS Controller + ScreenSpec (Фаза 1)**

* AddRow стане першим екраном нової архітектури
* І ми більше не будемо мати “Esc то працює, то ні”
* Не буде подвійних кнопок
* Не буде різних modal engine
