План 7 — Валідація напрямку та альтернативи міграції SWS ядра

Контекст
- Плани 1–6 зафіксували правильний перехід від локальних патчів до архітектурної централізації.
- Критична проблема: відсутність єдиного контролера модалок (SWS/SettingsWindow + UI.modal + injected overlay).
- Додатковий пріоритет: Debug має стати незалежним інструментом діагностики навіть при частковому падінні UI.

1) Чи правильний обраний напрямок?
Коротко: так, напрямок правильний.

Чому це правильно:
1. Відмова від patch-by-patch підходу зменшує системний борг.
2. Централізація модалок у єдиному SWS Core прибирає конфлікти стеку, клавіатури й фокусу.
3. Декларативний ScreenSpec дає прогнозованість і керованість тестами.
4. Розділення UI та ядра спрощує міграцію build/dist у майбутніх фазах.
5. Винесення правил кнопок/Enter/Esc у контролер прибирає дублювання в екранах.

Ризики поточного напрямку:
- Надто велика фаза міграції може зупинити релізи.
- Можлива деградація UX при одночасному перенесенні багатьох екранів.
- Без контрактних тестів ScreenSpec ризик регресій лишається високим.

Висновок:
- Напрямок вірний стратегічно.
- Потрібна більш дрібна тактика: короткі вертикальні інкременти + жорсткі debug-gates.

2) Альтернативи для полегшення міграції

Альтернатива A (рекомендована): Strangler Fig + Adapter Layer
Ідея:
- Нове SWS ядро запускається поруч зі старою системою.
- Усі старі виклики UI.modal/SettingsWindow йдуть через adapter API.
- Кожен екран переводиться окремо, без Big-Bang.

Плюси:
- Найменший ризик падіння всього UI.
- Можна швидко відкотити конкретний екран.
- Прозорий прогрес через debug-метрики: migrated/not migrated.

Мінуси:
- Тимчасово є додатковий шар сумісності.
- Потрібна дисципліна, щоб не накопичити adapter назавжди.

Альтернатива B: Feature Flags per Screen
Ідея:
- Для кожного екрану окремий прапор: old/new render path.
- Перемикання доступне у Debug Center.

Плюси:
- A/B порівняння поведінки у runtime.
- Швидке виявлення деградацій без редеплою.

Мінуси:
- Подвоєння перевірок на період міграції.
- Складніший QA matrix.

Альтернатива C: Big-Bang Rewrite (не рекомендовано)
Ідея:
- Повна заміна модального шару за один цикл.

Плюси:
- Швидше прибирається legacy-код на папері.

Мінуси:
- Максимальний ризик простою.
- Висока ціна помилки.
- Втрата керованості rollback.

Рекомендація:
- Прийняти альтернативу A як базову.
- Додати з B тільки прапори для критичних екранів (Settings/Transfer/Debug).

3) Оновлена тактика впровадження (мікрокроки)

Фаза P0 — Debug-first фундамент (обов'язково перед UI-міграцією)
P0.1: Додати незалежний debug launcher у shell сторінки.
- Кнопка викликає lightweight debug panel без залежності від SWS.
- Показує boot status, loading errors, активні listeners, останній runtime exception.

P0.2: Ввести єдиний debug event bus.
- Кожний модуль логуватиме lifecycle: open/close/render/error.
- Формат подій стабільний (timestamp, source, event, payload).

P0.3: Ввести debug gates на кожен крок міграції.
- Gate-pass умова: екран відкривається, Esc стабільний, primary button policy валідна, помилок у bus=0.

Фаза P1 — SWS Core contracts
P1.1: ScreenSpec schema + runtime validation.
P1.2: SWS Controller (stack, back, close, active screen).
P1.3: Button policy engine (primary/secondary/hidden).
P1.4: Keyboard policy engine (Esc/Enter/focus).

Фаза P2 — Вертикальна міграція екранів
Черговість (від найменш ризикового до найскладнішого):
1. AddRow
2. QuickNav
3. Backup
4. Transfer
5. Row actions
6. Debug screen
7. Settings stacks

Фаза P3 — Демонтаж legacy
- Прибрати прямі виклики UI.modal.
- Прибрати injected overlays.
- Закрити adapter шар після 100% міграції.

4) Definition of Done (DoD) для кожного екрану
Екран вважається мігрованим тільки якщо:
1. Відкривається через SWS.openScreen(spec,...)
2. Кнопки управляються controller policy
3. Esc/Enter/focus повністю через controller
4. Немає локальних global listeners у екрані
5. Є debug telemetry + smoke кнопка в Debug Center
6. Є rollback (feature flag або adapter fallback)

5) Як це узгоджується з motivation
- Принцип мінімального кроку: міграція екран-за-екраном.
- Принцип повної перевірки: debug gates перед merge.
- Принцип відсутності дублювання: поетапне вимкнення legacy після підтвердження.
- Принцип стабільності: rollback на рівні екрану, а не всього застосунку.

6) Підсумок рішення
- Вектор (централізоване SWS ядро) обрано правильно.
- Найкраща практична стратегія: Strangler + Adapter, debug-first, контрактні правила ScreenSpec.
- Це знижує ризики, зберігає темп і забезпечує керовану міграцію без глобального падіння UI.
